# Mini NPU Simulator

## 1. 프로젝트 개요

Mini NPU Simulator는 입력 패턴과 필터를 위치별로 곱한 뒤 모두 더하는 MAC(Multiply-Accumulate) 연산을 직접 구현하여, 입력 패턴이 `Cross`에 더 가까운지 `X`에 더 가까운지 판별하는 Python 콘솔 애플리케이션이다.

이 프로젝트는 단순히 점수만 계산하는 데서 끝나지 않고, 아래 기능을 하나의 콘솔 프로그램으로 통합하는 것을 목표로 했다.

1. 3×3 필터와 패턴을 직접 입력받아 판정하는 사용자 입력 모드
2. `data.json`을 읽어 여러 크기의 패턴을 일괄 판정하는 JSON 분석 모드
3. 라벨 정규화와 epsilon 기반 동점 처리
4. 크기별 MAC 연산 성능 분석
5. 실패 원인 및 시간 복잡도 분석 리포트 작성

> 참고: 과제 본문에는 `data.json` 구조 요구사항이 포함되어 있었지만 실제 파일은 별도 제공되지 않아, 요구사항에 맞는 샘플 `data.json`을 프로젝트 루트에 배치한 뒤 모드 2를 검증했다.

## 8. 핵심 구현 요약

### 8-1. 라벨 정규화

- `+`는 `Cross`, `x`는 `X`로 변환했다.
- 필터 키의 `cross`, `x`도 각각 `Cross`, `X`로 통일했다.
- 프로그램 내부 비교는 항상 표준 라벨 `Cross`, `X` 기준으로 수행했다.

라벨 정규화는 `+`, `cross`, `x`처럼 서로 다른 표현을 내부 표준 라벨인 `Cross`, `X`로 통일하는 단계다.  
이 로직을 `normalize_label()`로 따로 뺀 이유는 비교 기준을 한 곳에서만 관리하기 위해서다.

분리하지 않고 각 위치에서 직접 처리하면 다음 문제가 생긴다.

- 어떤 곳은 `+`를 `Cross`로 바꾸고, 어떤 곳은 그대로 둘 수 있다.
- 필터 키 처리와 expected 처리 방식이 달라질 수 있다.
- 출력 로직과 비교 로직이 서로 다른 라벨 체계를 사용할 수 있다.

반대로 별도 함수로 분리하면 장점이 크다.

1. 단일 책임  
   - 라벨 표현 통일이라는 한 가지 역할만 맡긴다.

2. 재사용성  
   - 필터 키 정규화, expected 정규화, 출력 전 정리 모두 같은 함수를 쓸 수 있다.

3. 유지보수성  
   - 나중에 새 라벨을 추가하더라도 `normalize_label()`만 수정하면 된다.

4. 일관성  
   - PASS/FAIL 비교, 콘솔 출력, 내부 저장 라벨이 모두 같은 기준을 사용하게 된다.

즉 `normalize_label()`를 별도 함수로 분리한 이유는 “정규화 규칙을 코드 전체에 흩뿌리지 않고, 하나의 진입점에서 관리하기 위해서”라고 설명할 수 있다.

### 8-2. MAC 연산

- `pattern[i][j] * filter[i][j]`를 모든 위치에서 수행한 뒤 누적 합계를 구하는 방식으로 구현했다.
- NumPy 같은 외부 라이브러리는 사용하지 않고, 중첩 반복문으로 직접 구현했다.

MAC 연산은 사용자 입력 모드와 JSON 분석 모드 모두에서 동일한 핵심 계산 함수로 동작하도록 설계했다.  
전체 흐름은 아래 순서다.

1. 입력 단계  
   - 모드 1에서는 `read_matrix_from_console()`가 3×3 패턴과 필터를 입력받는다.
   - 모드 2에서는 `load_json_file()`와 `analyze_single_pattern()`이 JSON의 `input` 배열을 읽는다.

2. 검증 단계  
   - `parse_matrix_row()` 또는 `validate_square_matrix()`가 행/열 개수와 숫자 형식을 확인한다.
   - 여기서 입력이 잘못되면 이후 계산으로 넘기지 않는다.

3. 위치별 곱셈 단계  
   - `mac(pattern, filt)` 함수에서 중첩 반복문으로 모든 좌표 `(i, j)`를 순회한다.
   - 각 위치에서 `pattern[i][j] * filt[i][j]`를 계산한다.

4. 누적 합산 단계  
   - 각 위치의 곱셈 결과를 `total += pattern[i][j] * filt[i][j]` 형태로 누적한다.
   - 이 누적값이 곧 해당 필터에 대한 유사도 점수가 된다.

5. 점수 반환 단계  
   - `mac()` 함수는 최종 누적합 `total`을 반환한다.
   - 이후 `judge_scores()`가 Cross 점수와 X 점수 또는 A 점수와 B 점수를 비교한다.

즉 코드 구조 기준으로 보면, 입력/검증 함수와 연산 함수가 분리되어 있고, `mac()`는 오직 “같은 위치 값끼리 곱해서 전부 더한 점수”만 계산하는 순수 계산 함수로 두었다.  
이렇게 분리하면 입력 방식이 달라도 MAC 계산 로직은 재사용할 수 있고, 테스트도 쉬워진다.

이 과제에서 필터는 “기준 패턴”이고, 패턴 입력은 “검사 대상”이다.  
MAC 연산은 같은 위치의 값끼리 곱한 뒤 모두 더하므로, 필터와 입력이 중요한 위치에서 많이 겹칠수록 누적합이 커진다.

예를 들어 0과 1 기반 패턴에서는 다음과 같이 해석할 수 있다.

- `1 × 1 = 1`  
  → 필터가 중요하다고 표시한 자리와 입력이 실제로 일치한 경우
- `1 × 0 = 0`, `0 × 1 = 0`  
  → 해당 위치는 겹치지 않은 경우
- `0 × 0 = 0`  
  → 중요하지 않은 자리

따라서 누적합이 크다는 것은 “필터가 강조하는 위치와 입력이 더 많이 겹친다”는 뜻이고, 이 과제에서는 그 값을 유사도 점수로 해석한다.  
즉 점수가 높을수록 해당 필터와 더 비슷한 모양이라고 판단할 수 있다.

### 8-3. 동점 처리 정책

- 부동소수점 오차로 인해 점수가 거의 같은 경우가 발생할 수 있으므로 단순 `==` 비교 대신 epsilon 정책을 사용했다.
- `abs(score_cross - score_x) < 1e-9`이면 `UNDECIDED`로 처리했다.

부동소수점 수는 컴퓨터 내부에서 2진수 기반으로 표현된다.  
이때 10진수에서는 단순해 보이는 수라도 내부 표현 과정에서 정확히 떨어지지 않을 수 있고, 여러 번의 연산을 거치면 아주 미세한 오차가 누적될 수 있다.

예를 들어 사람이 보기에는 같아 보이는 두 값이 실제 메모리 값에서는 아래처럼 다를 수 있다.

- `0.9000000000000000`
- `0.8999999999999999`

이 두 값을 단순 `==` 또는 `>` 비교만으로 처리하면, 사실상 동점인 경우를 억지로 승패로 나눌 수 있다.  
그래서 `abs(score_cross - score_x) < 1e-9` 같은 허용오차 기준을 두고, 차이가 아주 작으면 같은 값으로 간주한다.

즉 epsilon 정책이 필요한 이유는 다음과 같다.

1. 부동소수점 표현의 미세한 흔들림을 흡수하기 위해
2. 사실상 동점인 케이스를 오판정하지 않기 위해
3. 비교 정책을 코드상에서 명시적으로 고정하기 위해

이 과제에서는 그 결과를 `UNDECIDED`로 처리하도록 설계했다.

동점 자체를 처리하는 기준은 둘 다 epsilon 기반이라는 점에서 같다.  
차이는 “출력 의미”와 “요구사항의 관점”에 있다.

- mode1: 임의의 두 입력 필터 비교 → A/B/판정 불가
- mode2: 표준 필터 비교 + expected 대조 → Cross/X/UNDECIDED

모드 1은 사용자가 직접 필터 A와 필터 B를 입력하는 구조다.  
여기서 프로그램은 두 필터가 실제로 `Cross`, `X`인지 보장할 수 없다.  
즉 모드 1에서는 필터의 의미보다 “첫 번째 입력 필터 vs 두 번째 입력 필터”라는 비교가 중요하다.

반면 모드 2는 `data.json`에서 `Cross` 필터와 `X` 필터를 명시적으로 읽는다.  
여기서는 비교 대상 자체가 의미를 가진 표준 라벨이므로, 판정 결과도 표준 라벨 기준으로 유지하는 것이 자연스럽다.

### 8-4. 성능 측정 방식

- I/O와 파일 읽기 시간을 제외하고, 연산 함수 호출 구간만 10회 반복 측정했다.
- 평균 시간은 ms 단위로 출력했다.
- 성능 표에는 `크기(N×N) / 평균 시간(ms) / 연산 횟수(N²)`를 함께 표기했다.

시간 측정은 “프로그램 전체 실행 시간”이 아니라 “MAC 연산 자체가 얼마나 걸리는가”를 보기 위한 것이므로, 입력/출력과 파일 읽기 시간을 의도적으로 제외했다.

내가 잡은 경계는 다음과 같다.

- 포함한 구간:
  - `mac(pattern, filt)` 또는 이를 포함한 비교 함수 호출 구간
- 제외한 구간:
  - `input()`으로 사용자 입력받는 시간
  - `print()`로 화면에 출력하는 시간
  - `json.load()`로 파일을 읽는 시간
  - 루프 바깥의 메뉴 처리 시간

예를 들어 `measure_average_ms()` 내부에서는 아래와 같은 경계를 잡는다.

```python
start = time.perf_counter()
func(*args)
end = time.perf_counter()
```

즉 `start`는 연산 함수 호출 직전에 찍고, `end`는 연산 함수가 끝나자마자 찍는다.  
이렇게 해야 성능 표가 “사용자의 타이핑 속도”나 “콘솔 출력량”이 아니라, 순수한 계산 비용을 보여주게 된다.

### 8-5. JSON 매칭 구조

JSON 분석 모드에서는 `patterns`의 키가 `size_{N}_{idx}` 형태라고 가정했다.  
예를 들어 `size_13_1`은 “13×13 크기의 첫 번째 패턴”이라는 뜻이다.

이 구조를 사용한 이유는 패턴 데이터 안에 크기 정보를 별도 필드로 중복 저장하지 않고, 키 이름 자체에서 크기를 바로 읽어낼 수 있기 때문이다.  
구현 순서는 다음과 같다.

1. `extract_size_from_pattern_key(pattern_key)`에서 문자열을 `_` 기준으로 분리한다.
2. `size_13_1`이면 두 번째 토큰인 `13`을 꺼내 정수로 변환한다.
3. 추출한 `N`을 기준으로 `filters_by_size[N]` 또는 `size_N` 필터 그룹을 선택한다.
4. 선택한 필터와 패턴 입력이 모두 `N×N`인지 `validate_square_matrix()`로 다시 검증한다.
5. 검증 통과 시에만 `Cross` 필터와 `X` 필터 각각에 대해 MAC 점수를 계산한다.

이렇게 설계한 장점은 다음과 같다.

- 패턴마다 어떤 필터를 써야 하는지 자동으로 결정할 수 있다.
- 5×5, 13×13, 25×25처럼 크기가 다른 케이스를 하나의 루프로 처리할 수 있다.
- 잘못된 키 형식이나 크기 불일치를 케이스 단위 FAIL로 분리할 수 있다.

즉 “패턴 키 → 크기 추출 → 맞는 필터 선택” 흐름을 기준으로 설계하면, JSON 분석 모드의 자동 채점 구조가 단순해진다.

## 11. 성능 분석

### 11-1. 성능 표

```text
크기        평균 시간(ms)           연산 횟수(N^2)
---------------------------------------------
3x3       0.001486            9
5x5       0.002579            25
13x13     0.012216            169
25x25     0.041550            625
```

### 11-2. 해석

MAC 연산은 입력 패턴과 필터의 모든 위치를 한 번씩 순회하며 곱셈과 덧셈을 수행한다.  
따라서 입력 크기가 `N×N`일 때 기본 연산 횟수는 `N²`이 된다.

실제 측정 결과를 보면:

- 3×3은 9회
- 5×5는 25회
- 13×13은 169회
- 25×25는 625회

의 위치 계산이 필요했다. 평균 시간도 크기가 커질수록 전체적으로 증가하는 경향을 보였으므로, 이 연산의 시간 복잡도는 `O(N²)`로 해석할 수 있다.

즉, 패턴 크기가 커질수록 처리해야 하는 곱셈-덧셈 횟수가 급격히 늘어나며, 그에 따라 실행 시간도 증가한다는 점을 성능 표로 확인할 수 있었다.

패턴과 필터가 둘 다 `N×N` 행렬이면, MAC 연산은 모든 위치 `(i, j)`를 한 번씩 방문해야 한다.  
행이 `N`개이고, 각 행마다 열이 `N`개 있으므로 총 방문 횟수는 `N × N = N²`이다.

즉 반복문 구조 자체가 다음과 같은 중첩 루프이므로 연산량은 입력 크기의 제곱에 비례한다.

```python
for i in range(size):
    for j in range(size):
        total += pattern[i][j] * filt[i][j]
```

실측 결과도 크기가 커질수록 평균 시간이 증가하는 방향을 보였기 때문에, 이 구현의 시간 복잡도는 `O(N²)` 경향을 가진다고 설명할 수 있다.  
완전히 정확히 비례하지는 않더라도, 전체 경향은 “칸 수가 늘어날수록 계산 시간이 증가한다”는 구조와 일치한다.

## 12. 결과 리포트

이번 과제에서 가장 중요한 구현 포인트는 라벨 정규화, epsilon 기반 비교, 그리고 케이스 단위 예외 처리였다.

첫째, `data.json`의 `expected` 값과 필터 키는 그대로 비교하지 않고 먼저 표준 라벨인 `Cross`, `X`로 정규화했다.  
이 과정을 넣지 않으면 `+`, `cross`, `x`처럼 표기 방식이 다른 데이터가 들어왔을 때, 같은 의미인데도 문자열 불일치로 FAIL이 발생할 수 있다.

둘째, 점수 비교는 단순 `==`가 아니라 `abs(score_cross - score_x) < 1e-9` 기준으로 처리했다.  
부동소수점 연산에서는 사람이 보기엔 같아 보여도 내부적으로 아주 미세한 차이가 생길 수 있기 때문이다.  
이 정책을 넣지 않으면 사실상 동점인 케이스를 억지로 `Cross` 또는 `X`로 판정해 오답을 만들 위험이 있다.

셋째, JSON 분석 모드에서는 특정 케이스 하나가 잘못돼도 프로그램 전체가 중단되지 않도록 케이스 단위로 예외를 처리했다.  
이 방식으로 스키마 오류, 크기 불일치, 키 형식 오류를 개별 FAIL로 기록할 수 있게 했다.

넷째, MAC 연산은 모든 위치를 한 번씩 순회하며 곱셈과 덧셈을 수행하므로 입력 크기가 `N×N`일 때 기본 연산 횟수는 `N²`이다.  
실제 실행 결과에서도 3×3은 9회, 5×5는 25회, 13×13은 169회, 25×25는 625회의 위치별 계산이 필요했고, 평균 시간도 전체적으로 증가했다.  
따라서 이 연산의 시간 복잡도는 `O(N²)`로 해석할 수 있다.

이번 실행에서는 총 8개 패턴 중 7개가 PASS였고, `size_13_1`만 FAIL이 발생했다.  
이 케이스는 `Cross 점수`와 `X 점수`가 둘 다 `22.500000`으로 동일하게 계산되어 프로그램이 `UNDECIDED`를 반환했다.  
하지만 데이터의 expected 값은 `X`였기 때문에 최종 결과는 FAIL로 기록되었다.

이 결과는 구현 오류라기보다 현재 판정 정책의 사례이다.  
동점일 때 승자를 강제로 선택하지 않고 `UNDECIDED`로 처리하도록 설계했기 때문에, 기대 라벨이 하나로 지정된 데이터와는 충돌할 수 있다.

즉 이번 FAIL의 원인은 다음 세 가지 중 계산 로직 오류가 아니라,  
**수치 비교 정책과 데이터 기대값 사이의 차이**에 가깝다.

- 데이터/스키마 문제: 아님
- 로직 문제: 아님
- 수치 비교 정책 문제: 해당

따라서 본 구현은 과제 요구사항에 맞게 동점 처리 정책을 적용했고, 그 결과가 데이터 기대값과 충돌하면서 FAIL이 1건 발생했다고 정리할 수 있다.

만약 `data.json`에 새 라벨 `o`가 추가된다면, 확장은 아래 순서로 하는 것이 가장 안전하다.

1. 정규화 확장  
   - 먼저 `normalize_label()`에 `'o' -> 'O'` 또는 새 표준 라벨 매핑을 추가한다.
   - 이 단계가 먼저여야 나머지 비교 로직이 같은 표준 값을 사용하게 된다.

2. 필터 구조 확장  
   - `filters` 로드 로직에서 `Cross`, `X` 외에 새 필터 라벨도 읽도록 수정한다.
   - 필요한 경우 `filters_by_size[size]` 구조를 2개 고정이 아니라 다중 라벨 딕셔너리로 바꾼다.

3. 판정 로직 확장  
   - 기존처럼 두 점수만 비교하는 구조가 아니라, 여러 라벨 점수 중 최대값을 찾는 구조로 일반화한다.
   - 동점 규칙도 “최대 점수가 둘 이상이면 UNDECIDED”처럼 확장한다.

4. 출력 로직 확장  
   - 점수 출력 시 `Cross`, `X`만 고정 출력하지 말고, 모든 라벨 점수를 순회 출력하도록 바꾼다.

5. 테스트 확장  
   - 새 라벨이 포함된 정상 케이스
   - 새 라벨과 기존 라벨 간 동점 케이스
   - 정규화 실패 케이스
   - 필터 누락 케이스
   를 추가한다.

즉 확장 순서는 **정규화 → 필터 로드 → 판정 → 출력 → 테스트**가 가장 안전하다.  
가장 먼저 정규화를 수정해야 이후 단계가 같은 기준을 공유할 수 있기 때문이다.

또한 1000×1000 패턴은 총 1,000,000개의 위치를 가진다.  
필터 하나와의 MAC 연산만 해도 백만 번의 위치 계산이 필요하고, 필터가 여러 개면 그만큼 반복된다.  
파이썬 순수 반복문으로 처리하면 실행 시간 증가, 메모리 사용량 증가, 큰 JSON 파일 파싱 비용 증가, 로그 처리 부담 증가 같은 문제가 생길 수 있다.

가장 큰 병목은 `mac()` 내부의 중첩 반복문이다.  
즉 모든 좌표를 Python 레벨에서 직접 순회하면서 곱셈과 덧셈을 수행하는 부분이 핵심 병목이다.

그 다음 병목은 아래와 같다.

- 큰 2차원 배열을 메모리에 보관하는 비용
- 큰 JSON 파일을 읽고 파싱하는 비용
- 여러 필터를 연속 비교할 때 같은 패턴을 반복 순회하는 비용

이때 고려할 수 있는 최적화 방향은 다음과 같다.

1. 메모리 접근 단순화  
   - 보너스 과제처럼 2차원 배열을 1차원 배열로 평탄화하여 인덱스 계산을 단순화한다.
   - 중첩 인덱싱 `matrix[i][j]`보다 연속된 1차원 접근이 캐시 친화적일 수 있다.

2. 불필요한 검증/출력 최소화  
   - 대형 입력에서는 반복되는 `print()`나 중복 검증을 줄인다.
   - 성능 측정 시 특히 I/O를 철저히 제외한다.

3. 반복 구조 단순화  
   - 지역 변수 바인딩, 길이 캐싱 등 파이썬 인터프리터 오버헤드를 줄이는 방식으로 미세 최적화를 할 수 있다.
   - 예를 들어 반복문 안에서 `len(pattern)`, `pattern[i]`, `filt[i]`를 계속 다시 참조하기보다, `size = len(pattern)`처럼 길이를 미리 저장하고 `pattern_row`, `filt_row`를 지역 변수로 받아 두면 불필요한 조회를 줄일 수 있다.

4. 필터 수 확장 대비 구조 개선  
   - 동일 패턴을 여러 필터에 비교해야 한다면, 패턴 순회를 최소화하거나 필터 표현 방식을 단순화하는 방향을 검토한다.
   - 예를 들어 `cross_score = mac(pattern, cross_filter)`, `x_score = mac(pattern, x_filter)`, `o_score = mac(pattern, o_filter)`처럼 작성하면 같은 패턴을 필터 수만큼 반복 순회하게 된다.
   - 입력이 작을 때는 큰 문제가 없지만, 1000×1000처럼 커지면 이 반복 비용이 커질 수 있다.
   - 따라서 패턴을 한 번만 순회하면서 여러 필터 점수를 함께 누적하는 구조로 바꾸면, 큰 입력에서 더 효율적인 방향으로 개선할 수 있다.

5. 더 큰 규모에서는 구현 방식 재검토  
   - 이 과제는 외부 라이브러리 금지라 순수 Python 반복문이 맞지만, 실제 대규모 처리라면 벡터화 라이브러리나 저수준 최적화가 필요하다.

즉 1000×1000 규모에서의 핵심 병목은 “Python 수준의 위치별 순회 자체”이며, 최적화의 첫 출발점은 데이터 접근 패턴 단순화와 불필요한 부가 비용 제거라고 설명할 수 있다.

## 13. 트러블슈팅

### 문제 1. 3×3 입력 중 열 개수가 맞지 않아 예외가 발생할 수 있었음

- 문제: 사용자가 `0 1`처럼 숫자를 2개만 입력할 수 있음
- 원인 가설: 행별 길이 검증이 없어서 잘못된 입력이 그대로 저장될 수 있음
- 확인: `parse_matrix_row()`에서 입력 길이를 검사하지 않으면 이후 연산에서 인덱스 오류가 발생할 수 있었음
- 해결: 각 줄 입력 시 기대 개수와 실제 개수를 즉시 비교하고, 오류 메시지를 출력한 뒤 재입력을 유도함

### 문제 2. JSON 비교 시 같은 의미의 라벨인데 FAIL이 날 수 있었음

- 문제: `+`, `cross`, `x` 같은 라벨 표현 차이 때문에 PASS/FAIL 비교가 흔들릴 수 있음
- 원인 가설: `expected` 값과 필터 키를 문자열 그대로 비교하면 표기 차이를 처리하지 못함
- 확인: 비교 전에 `normalize_label()`을 거치도록 하니 기준이 `Cross`, `X`로 통일됨
- 해결: 모든 라벨을 먼저 표준 라벨로 정규화한 뒤 판정 및 PASS/FAIL 비교를 수행함
